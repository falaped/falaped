<objective>
Enable Row Level Security with ownership policies on every `public.*` data table on the LIVE production database, rolled out incrementally one domain at a time via the Supabase MCP server (D-01, D-02). Each migration enables RLS and creates all CRUD policies atomically (never enable-only — that locks the table immediately), ships with a ready reversal script (D-05), and is validated against the app's flows before moving to the next domain. SEC-02 is the only requirement here, but it is the most operationally dangerous work in the phase.

Purpose: RLS is the database-layer backstop that makes the data layer safe to expose to patients — even if an application query forgets an ownership filter, Postgres enforces tenant isolation. This depends on Plan 01 (the delete actions must already pass `profile_id` so RLS does not break them) and benefits from Plan 02's CI/build gate.

Output: five timestamped RLS migrations applied to the live DB via MCP; one reversal-scripts file in `supabase/docs/`; a verified `rowsecurity = true` for every public data table.

CRITICAL OPERATIONAL CONSTRAINTS (from CONTEXT.md, non-negotiable):
- D-03: ONE live production project. No staging, no DB branch.
- D-04: HARD CONSTRAINT — no existing data may be deleted or altered. Migrations are additive only (enable RLS, create policies, create indexes). NEVER DML on existing rows, NEVER drop tables/columns.
- D-05: If a policy breaks an app flow, immediate rollback via the ready reversal script (data untouched), then investigate offline.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated doctor → Postgres | Any query (including a future one that forgets a `profile_id` filter) crosses into the DB; RLS is the last line of defense |
| Migration apply → live production data | DDL applied to real patient data; an enable-only or wrong-column policy can deny all access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-07 | Tampering / Information Disclosure | All public data tables lacking RLS — application-only tenant isolation, one missing filter exposes/destroys another doctor's data | mitigate | Enable RLS + ownership policies on every public data table, anchored on `profiles.auth_user_id = auth.uid()` (Task 1, Task 3). High severity — DB-layer enforcement regardless of app code path (SEC-02). |
| T-01-08 | Denial of Service (self-inflicted) | Enabling RLS without policies = instant deny-all lockout on a live table | mitigate | Every migration enables RLS and creates all CRUD policies atomically; post-apply read smoke test per domain; ready reversal script for immediate rollback (Task 1, Task 3, D-05). High severity. |
| T-01-09 | Elevation of Privilege | `security definer` helper `auth_user_phone()` with mutable search_path could be hijacked | mitigate | Helper declared `security definer` with `set search_path = ''` (matches existing trigger-function convention); only created if the live cases data requires the two-hop (Task 1). Medium severity. |
| T-01-10 | Denial of Service | RLS policy breaks account-deletion cascade (service_role unexpectedly restricted) | accept | `service_role` has `bypassrls`; policies target `to authenticated` only, so the admin delete path is unaffected. Verified post-apply (Task 3, Pitfall 2). Low residual risk. |
</threat_model>
