<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → build | Dependency versions resolved at install time; `latest` tag lets an unreviewed breaking version enter on any install |
| Git push → CI runner | Untrusted branch contents run in CI; the lockfile is the integrity anchor for what gets installed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Tampering (supply chain) | `@supabase/ssr` / `@supabase/supabase-js` pinned to `latest` — most security-sensitive deps (auth/session/data) | mitigate | Pin to explicit semver ranges + enforce `yarn install --frozen-lockfile` in CI so a divergent or breaking version fails the build (Task 1, Task 2). Medium severity. |
| T-01-SC | Tampering | npm installs during this plan (`yarn install` to update lockfile) | mitigate | No NEW packages are installed — only existing official Supabase packages are re-pinned (RESEARCH Package Legitimacy Audit: both Approved, multi-year registry history, verified Supabase-org repos). No `[ASSUMED]`/`[SUS]` packages, so no blocking-human checkpoint required. |
| T-01-06 | Tampering (double-apply) | `run_prescriptions_manual.sql` in `migrations/` could be applied as a real migration against the live DB | mitigate | Move out of `migrations/` to `supabase/docs/` with no timestamp prefix (Task 1, D-11) so `supabase db push` cannot pick it up. Low severity. |
</threat_model>
