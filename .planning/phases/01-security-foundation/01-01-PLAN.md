<objective>
Close the live IDOR in prescription and medical-certificate deletes (SEC-01), remove the service-role admin client from user-triggered delete paths (SEC-03), and convert the per-item bulk-delete loops into single batched DB+storage operations (SEC-04). All work is server-side code in `modules/` and `actions/` plus their test specs — zero database changes, zero production risk.

Purpose: This is the highest-severity finding in the security audit — an authenticated doctor can currently delete another doctor's prescription/certificate (DB row + PDF) by supplying its UUID, and the bulk path does so with god-mode service-role privileges. Fixing the code first (no DB risk) is the prerequisite for validating the RLS rollout (Plan 03) against working app flows.

Output: Four module files (two modified, two new) with ownership-scoped deletes; four action files threading `profile.id` and dropping the admin client; four test specs proving ownership no-op and single-call batching.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Doctor browser → Server Action | Untrusted UUIDs (prescription/certificate ids, storage paths) cross into server actions; a session exists but the supplied id may belong to another profile |
| Server Action → modules → Supabase | Server-side; the ownership filter and client choice (user vs admin) are enforced here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering / Spoofing | `modules/prescriptions/delete-prescription.ts`, `modules/medical-certificates/delete-medical-certificate.ts` (delete by id with no owner check — live IDOR) | mitigate | Add `.eq("profile_id", profileId)` to the DB delete and thread `profile.id` from the action (Task 2). Unauthorized ids become a no-op (SEC-01). High severity — explicit mitigation in Task 2, verified by ownership specs. |
| T-01-02 | Elevation of Privilege | `actions/*/delete-*.ts` instantiating `createAdminClient()` (service-role bypasses all RLS) in user-triggered paths | mitigate | Remove `createAdminClient` import + usage from all four delete actions; rely on user-scoped client + existing storage RLS (Task 2, Task 3). High severity. Verified by grep gate excluding delete-account.ts (SEC-03). |
| T-01-03 | Tampering (partial state) | `actions/*/delete-*-bulk.ts` per-item loop leaving rows 1..N-1 deleted on failure at item N | mitigate | Single `.in("id", ids)` DB delete + single batched `storage.remove` (Task 3). Atomic at Postgres level; storage failure logs orphan instead of aborting (SEC-04). Medium severity. |
| T-01-04 | Information Disclosure | Orphan PDF left in storage after DB row deleted (storage-remove failure) | accept | DB-first ordering means a storage failure leaves an orphan PDF in the owner's own bucket folder (storage RLS still scopes it to that profile) — no cross-tenant exposure. Logged for async cleanup. Low severity; documented rationale per D-07. |
</threat_model>
