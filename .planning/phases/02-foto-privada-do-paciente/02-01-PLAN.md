## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous client → Supabase Storage object | An unauthenticated `curl` can hit the object URL directly; the bucket must reject it |
| authenticated doctor A → patient row / object of doctor B | Cross-tenant access attempt (app has NO table RLS by default — CONCERNS) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | `patient-photos` storage object accessed via unauthenticated `curl` | mitigate | Bucket `public=false` + storage RLS `(storage.foldername(name))[1] in (select id::text from public.profiles where auth_user_id = auth.uid())` — verified in Plan 03 curl test (success criterion 2) |
| T-02-02 | Elevation of Privilege | Cross-tenant object access (doctor B reads doctor A's photo) | mitigate | Path prefixed by `profile_id` + the same `foldername[1]` RLS predicate restricts every object to its owner |
| T-02-03 | Tampering | Migration not applied to live DB → false-positive verification | mitigate | [BLOCKING] schema-push task (Task 3) with explicit live-DB column/bucket/policy checks |
| T-02-SC | Tampering | npm install of `browser-image-compression` | transfer | Handled in Plan 02 via blocking package-legitimacy checkpoint; no installs in this plan |
