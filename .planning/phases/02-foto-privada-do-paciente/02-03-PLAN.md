<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous client → storage object | Direct unauthenticated `curl` to the object; must be rejected (verified here) |
| authenticated doctor → another doctor's photo/row | Cross-tenant delete attempt |
| "deleted" data → storage | Object must not survive a delete (LGPD) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | Repudiation / Privacy | Photo object orphaned after delete (sensitive minor data persists) | mitigate | `deletePatientPhoto` invoked before the row delete in `delete-patient.ts` AND on photo-remove; idempotent; verified via spec + the curl/lifecycle checkpoint (success criterion 3 / Pitfall 2) |
| T-02-10 | Information Disclosure | Unauthenticated object access | mitigate | Bucket `public=false` + storage RLS (Plan 01); VERIFIED here by the unauthenticated `curl` returning 400/403 (success criterion 2) |
| T-02-11 | Elevation of Privilege | Cross-tenant photo removal | mitigate | Remove action reads owner-scoped via `getPatientById(.., profile.id)`; `delete-patient` keeps `.eq("profile_id")`; paid gate enforced |
| T-02-12 | Information Disclosure | Accidental destructive click wiping sensitive data | mitigate | Mandatory `AlertDialog` confirmation before remove (Task 3) |
</threat_model>
