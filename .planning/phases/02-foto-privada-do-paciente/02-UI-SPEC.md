# Phase 2 — UI Design Contract: Foto Privada do Paciente

> Visual and interaction contract for the private patient photo feature. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** avatar com foto em 3 superfícies (hero, lista, cabeçalho do caso), upload de foto + checkbox de consentimento LGPD no formulário, e remoção da foto. Stack já 100% disponível no repo — esta fase **estende componentes existentes**, não cria primitivas novas.

- NÃO usar `getPublicUrl` nem bucket público. A foto é privada (`createSignedUrl`).
- NÃO permitir `image/svg+xml` no upload (vetor de XSS).

## Color

Tokens OKLCH de `app/globals.css`. **Sempre usar tokens — nunca hex/rgb em componentes** (regra `design-system`).

| Role | Token | Valor (light) | Usage |
|------|-------|---------------|-------|
| Dominant (60%) | `--background` / `bg-background` | `oklch(0.985 0 0)` | Fundo da página, fundo do avatar sem foto no hero |
| Secondary (30%) | `--card` / `--muted` (`bg-card`, `bg-muted/20`) | `oklch(1 0 0)` / `oklch(0.97 0 0)` | Cards (hero, caso), zona do header do hero (`bg-muted/20`), fallback do avatar na lista (`bg-primary/15`) |
| Accent (10%) | `--primary` / `bg-primary` | `oklch(0.7603 0.0916 255.41)` (azul suave) | Ver lista de reserva abaixo |
| Destructive | `--destructive` / `text-destructive` | `oklch(0.58 0.22 27)` (vermelho) | Erro de validação do upload; erro de consentimento; ação "Remover foto" e sua confirmação |

**Accent (`primary`) reservado exclusivamente para:**
- Estado marcado do checkbox de consentimento (`data-[state=checked]:bg-primary` — vigente no `Checkbox`).
- Iniciais do fallback do avatar (`text-primary` sobre `bg-primary/10` no hero, `bg-primary/15` na lista — vigente).
- Borda do checkbox não-marcado (`border-primary` — vigente no `Checkbox`).
- `focus-visible:ring-ring` (anel de foco azul) em input de foto, checkbox e botões.
- Botão primário "Salvar" do formulário (já existente — não é elemento novo desta fase).

NÃO usar accent para: texto de corpo, bordas de card decorativas, ícones informativos (esses usam `text-muted-foreground`), ou o botão "Remover foto" (esse é `destructive`/`ghost`).

**Estado de erro alarmante vs. gracioso (decisão de cor importante):**
- **URL assinada expirada / falha de carregamento da imagem:** NÃO é erro — o `AvatarFallback` (iniciais sobre `bg-primary/10|15`) cobre automaticamente. Nenhum vermelho, nenhum ícone de alerta. (Pitfall 1 do RESEARCH — fallback gracioso.)
- **Erro de validação no upload** (tipo/tamanho) e **consentimento não marcado:** aí sim usar `text-destructive` + mensagem.

## Copywriting Contract

Todas as strings em **PT-BR** (CLAUDE.md + skill). Tom: profissional, direto, orientado ao pediatra; "Responsável" nunca "mãe/pai".

### CTAs e labels

| Element | Copy |
|---------|------|
| Primary CTA (enviar/trocar) | **"Enviar foto"** quando não há foto; **"Trocar foto"** quando já existe uma |
| Label do campo | **"Foto do paciente"** |
| Hint sob o input | **"PNG, JPEG ou WebP. A imagem é otimizada automaticamente antes do envio."** |
| Botão de remover | **"Remover foto"** (ícone `Trash2` + texto; variant `ghost` com `text-destructive`) |

### Checkbox de consentimento (D-04 / D-05 / D-06 — BLOQUEANTE)

| Element | Copy |
|---------|------|
| Label do checkbox | **"Confirmo o consentimento do responsável para armazenar esta foto."** |
| Erro quando submete sem marcar | **"É necessário confirmar o consentimento do responsável para enviar a foto."** (`text-destructive text-sm` abaixo do checkbox) |

**Posicionamento e comportamento (interaction contract):**
- O checkbox fica **imediatamente abaixo** do input de foto e do preview, dentro da mesma seção "Identificação e contato" do `patient-form`. **Sem modal, sem tela dedicada** (D-04).
- O checkbox **bloqueia o envio**: enquanto não marcado, o botão "Enviar foto"/"Salvar" referente à foto fica **desabilitado** OU dispara a mensagem de erro acima ao tentar submeter. (Validação real é no Server Action com Zod — `consent === true` — não confiar só no client; Pitfall 4 do RESEARCH.)
- **Re-exigência ao trocar (D-06):** ao selecionar uma nova foto (substituição), o checkbox **reseta para desmarcado** e o consentimento é exigido novamente antes do envio. Cada envio grava `consent_at = now()`.

### Empty / estados do avatar

| Element | Copy / Comportamento |
|---------|----------------------|
| Empty state (sem foto) — hero/lista/caso | **Sem texto.** O `AvatarFallback` mostra as iniciais do nome (via `getPatientInitials`). Sem foto ≠ erro. |
| Empty state — área de upload no form (nenhuma foto ainda) | Avatar com iniciais (preview) + botão **"Enviar foto"** ao lado. Sem placeholder "arraste aqui" (input clássico, D-07). |
| Carregando (otimizando no cliente) | **"Otimizando imagem…"** (texto curto ao lado do botão; botão em estado `disabled`) |
| Enviando (upload em andamento) | Botão em **"Enviando…"** (`disabled`); usar o padrão `Salvando…` do projeto |
| Sucesso | Toast (`sonner`): **"Foto atualizada."** + o avatar passa a mostrar a nova imagem |

### Error states (upload)

| Cenário | Copy (`text-destructive` inline + toast) |
|---------|------------------------------------------|
| Tipo inválido | **"Tipo de arquivo não permitido. Use PNG, JPEG ou WebP."** (espelha `upload-profile-logo.ts`) |
| Tamanho acima do limite | **"Arquivo muito grande. Envie uma imagem de até 2 MB."** |
| Falha genérica de upload | **"Não foi possível enviar a foto. Tente novamente."** |
| Consentimento ausente | **"É necessário confirmar o consentimento do responsável para enviar a foto."** |

### Destructive — Remover foto (critério de sucesso 3: remove objeto + referência)

| Element | Copy |
|---------|------|
| Affordance | Botão **"Remover foto"** (`ghost`, `text-destructive`, ícone `Trash2`), visível apenas quando há foto, ancorado junto do preview no form |
| Confirmação | **`AlertDialog`** (já disponível no shadcn) — Título: **"Remover foto do paciente?"** / Descrição: **"A foto será excluída permanentemente do armazenamento. Esta ação não pode ser desfeita."** / Confirmar: **"Remover"** (`variant="destructive"`) / Cancelar: **"Cancelar"** |
| Sucesso | Toast: **"Foto removida."** + avatar volta para iniciais |

> A confirmação é obrigatória porque a exclusão é **destrutiva e irreversível** (remove o objeto do storage + a referência `photo_path`/consentimento). Sem confirmação, um clique acidental apagaria dado sensível.

## Avatar — Dimensões por superfície (Claude's Discretion, D-13)

Decisão registrada (respeitando o design vigente do dashboard; ver tamanhos já no código):

| Superfície | Componente | Tamanho | Classe | Origem |
|------------|-----------|---------|--------|--------|
| **Hero / identificação** | `patient-detail-hero.tsx` | 80px → 96px (`sm:`) | `h-20 w-20 sm:h-24 sm:w-24` | **Já vigente** — manter; apenas adicionar `<AvatarImage>` |
| **Miniatura na lista** | `patients-table.tsx` | 24px | `size="sm"` (size-6) | **Já vigente** — manter; adicionar `<AvatarImage>` por linha |
| **Cabeçalho do caso** | `case-patient-info.tsx` | 40px | `size` default (size-8) → recomendado subir para `h-10 w-10` para leitura confortável no card | **Novo** — o card atual NÃO tem avatar; adicionar ao lado do `CardTitle` |

**Forma e recorte (todos):**
- **Circular** com fallback (D-11) — `rounded-full` (padrão do `Avatar`).
- `AvatarImage` usa `object-cover` (já no componente) → **recorte centralizado**, adequado a fotos que incluam o médico junto da criança ("foto do médico com a criança", PROJECT.md). Não distorce.
- `alt` descritivo e PT-BR: **`Foto de {nome do paciente}`**.
- Avatar no hero e na lista deve manter a borda/anel sutil já vigente (`border border-border/80` no hero; `ring-1 ring-border/80` na lista) para separar a foto do fundo claro.

**Estados do avatar (contrato de interação, todas as superfícies):**

| Estado | Render |
|--------|--------|
| Com foto | `<AvatarImage src={signedUrl} alt="Foto de {nome}" />` sobre o fallback |
| Sem foto | Apenas `<AvatarFallback>` com iniciais (`getPatientInitials`) — comportamento atual preservado |
| Carregando signed URL | Sem skeleton dedicado: o fallback de iniciais já aparece até a `<img>` carregar (Radix mostra fallback enquanto a imagem não resolve) |
| Erro ao carregar / URL expirada | Radix volta automaticamente ao `AvatarFallback` (iniciais). **Sem estado de erro visível.** (Pitfall 1) |

## Upload — Layout no formulário (PHOTO-01)

Onde mora: seção **"Identificação e contato"** do `components/dashboard/patients/patient-form/patient-form-personal-section.tsx` (ou subcomponente novo `patient-form-photo-field.tsx` importado ali), acima ou ao lado dos campos de nome.

Estrutura visual (contrato):
```
[ Avatar preview 64–80px ]   [ input de arquivo clássico ]  ( "Otimizando…" / "Enviando…" )
                              [ Button "Enviar foto" / "Trocar foto" ]   [ ghost "Remover foto" (se há foto) ]
[ ☐  Confirmo o consentimento do responsável para armazenar esta foto. ]   ← checkbox bloqueante
[ text-destructive: erros de validação / consentimento ]                    ← só quando há erro
[ text-sm text-muted-foreground: "PNG, JPEG ou WebP. A imagem é otimizada automaticamente antes do envio." ]
```

- Input de arquivo é **clássico, sem atributo `capture`** (D-07): `accept="image/png,image/jpeg,image/webp"`.
- Ao escolher arquivo: mostrar **preview** no avatar imediatamente (object URL local), exibir "Otimizando imagem…" durante a compressão cliente (`browser-image-compression`), depois "Enviando…" durante o Server Action.
- Validação cliente espelha o module (tipo + tamanho ≤ 2 MB pós-compressão) com mensagens PT-BR idênticas; validação **autoritativa** é no action (Zod) + module.
