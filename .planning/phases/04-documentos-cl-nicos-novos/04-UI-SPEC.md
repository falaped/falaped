# Phase 4 — UI Design Contract

> Visual and interaction contract for the five new clinical-document surfaces (encaminhamento, pedido de exames, relatório médico, receituário em branco, biblioteca de orientações) + savable templates. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This is NOT a greenfield design.** Phases 1–3 shipped against an established system. Every new surface MUST mirror the existing prescription / medical-certificate analogs. Prescriptiveness beats creativity (RESEARCH "Key insight": the phase risk is *divergence*, not novelty). Where this spec says "match the analog", the analog is authoritative and this document only records the contract it already embodies.

## Spacing Scale

Declared values (all multiples of 4). Values below are the Tailwind utilities the existing analogs already use — reuse them; do not introduce new spacing.

| Token | Value | Tailwind | Usage (from existing analogs) |
|-------|-------|----------|-------------------------------|
| xs | 4px | `gap-1`, `mr-1` | Icon-to-count micro gaps |
| sm | 8px | `gap-2`, `p-2` | Compact chip/badge/inline spacing; icon+label (`mr-1.5` = 6px is the one existing exception, see below) |
| md | 16px | `gap-4`, `p-4`, `space-y-4` | Default field spacing, card padding on compact cards |
| lg | 24px | `gap-6`, `p-6` | **Page container gap (`flex flex-col gap-6`) and page padding (`p-6`) — the standard**; section padding |
| xl | 32px | `p-8` | Empty-state card padding (`rounded-xl border border-dashed p-8`); larger page padding |
| 2xl | 48px | `gap-12` | Major section breaks (rare in these list/form surfaces) |
| 3xl | 64px | — | Page-level spacing (not expected in this phase) |

Card content padding: `p-5` (20px) on list cards (`prescription-card.tsx`) or `p-6` on form cards — match the analog you clone. `p-5` is the established list-card value; keep it for parity even though 20 is not on the 8-scale (documented exception).

**Exceptions (pre-existing, keep for parity — do NOT "correct"):**
- `mr-1.5` (6px) icon→label gap on buttons — this is the shipped convention across prescription/certificate buttons; reuse it verbatim.
- `p-5` (20px) list-card padding — shipped in `prescription-card.tsx`; new document cards match it.

## Typography

Declared roles (4 sizes, 2 weights). These are the exact values the existing dashboard surfaces use (design-system skill + analog reads). Do not invent a new type scale.

| Role | Size | Weight | Line Height | Tailwind |
|------|------|--------|-------------|----------|
| Page heading (h1) | 24px | 600 (semibold) | 1.2 (tight) | `text-2xl font-semibold tracking-tight` |
| Body / value | 16px | 500 (medium) for emphasized values, else 400 | 1.5 | `text-base` / `font-medium` on primary values (e.g. date in card) |
| Label / secondary | 14px | 400 (regular) | 1.5 | `text-sm text-muted-foreground` (descriptions, field labels, table headers) |
| Caption / meta | 12px | 400 (regular) | 1.4 | `text-xs text-muted-foreground` (e.g. "PDF não disponível", stepper hints) |

**Weights: exactly two — regular (400) and semibold (600).** `font-medium` (500) is used sparingly on emphasized inline values (card primary line) and is the only permitted third weight, matching the shipped analog; treat 400/600 as the contract, 500 as the pre-existing emphasis exception.

**Header pattern (every new route page — from design-system skill, mandatory):**
```tsx
<div className="flex flex-col gap-6">
  <div>
    <h1 className="text-2xl font-semibold tracking-tight">[Título]</h1>
    <p className="text-sm text-muted-foreground">[Descrição breve]</p>
  </div>
  {/* conteúdo */}
</div>
```

## Color

60/30/10 split, expressed via the existing OKLCH token system (`app/globals.css`). Components reference tokens only.

| Role | Token | Value (light) | Usage |
|------|-------|---------------|-------|
| Dominant (60%) | `--background` / `bg-background` | `oklch(0.985 0 0)` | Page background, primary surface |
| Secondary (30%) | `--card` / `--muted` / `--secondary` | `oklch(1 0 0)` / `oklch(0.97 0 0)` | Cards, sidebar, table zebra (`even:bg-muted/50`), section surfaces |
| Accent (10%) | `--primary` / `bg-primary text-primary-foreground` | `oklch(0.7603 0.0916 255.41)` (soft blue) | See reserved list below |
| Destructive | `--destructive` / `text-destructive` / `variant="destructive"` | `oklch(0.58 0.22 27)` | Destructive actions + inline validation errors only |

**Accent (`--primary`) reserved for — explicit list, nothing else:**
- The single primary CTA per surface (the generate/save button: `Button` default variant)
- Active navigation/sidebar item (`bg-primary/10 text-primary` per design-system skill)
- Focus ring (`--ring`, primary-derived) and subtle card hover accent (`hover:border-primary/30` — the shipped `prescription-card` treatment)

Everything else (secondary buttons, "Baixar PDF", "Usar template", "Remover paciente", table row actions) uses `variant="outline"` / `variant="ghost"` — **neutral, not accent**. Urgency badges (DOC-01) are NOT accent-colored: use `Badge` `variant="secondary"` for Rotina, `variant="default"` for Prioritário, `variant="destructive"` for Urgente (reuse the shipped Badge; do not create new color tokens).

**Destructive (`--destructive`) reserved for:** delete/exclude actions (row + bulk), delete confirmation dialogs, the "Urgente" urgency badge, and inline field validation errors (`text-destructive text-sm` below the field, per forms rule). Never for informational or primary-action emphasis.

### Error states

- Inline field validation: `text-destructive text-sm` below the field, PT-BR via `zodErrorToUserMessage` (forms rule).
- Action failure: `toast.error(getFriendlyToastMessage(result.error))` — never surface raw `[DOMAIN]` errors.
- Pre-submit guards mirror the analog, e.g. exam request with no items: `toast.error("Adicione pelo menos um exame ao pedido.")`; referral missing specialty: `toast.error("Informe a especialidade ou serviço de destino.")`; report empty body: `toast.error("Escreva o corpo do relatório.")`.
- Session/paid-gate failures come back as PT-BR result strings (`Sessão não encontrada.` / `Perfil não ativo. Conclua a configuração da conta em Perfil.`) — surface via `toast.error` unchanged.

### Destructive confirmations (AlertDialog — match prescription/certificate table)

| Action | Dialog title | Confirm button (idle → busy) | Cancel |
|--------|--------------|------------------------------|--------|
| Delete one encaminhamento | `Excluir encaminhamento?` | `Excluir` → `Excluindo…` | `Cancelar` |
| Delete one pedido de exames | `Excluir pedido de exames?` | `Excluir` → `Excluindo…` | `Cancelar` |
| Delete one relatório médico | `Excluir relatório?` | `Excluir` → `Excluindo…` | `Cancelar` |
| Bulk delete (n selected) | `Excluir 1 {tipo}?` / `Excluir {n} {tipo}s?` | `Excluir selecionados` → `Excluindo…` | `Cancelar` |

Confirm button uses `variant="destructive"`. Success toast e.g. `Encaminhamento excluído.` / `Pedido de exames excluído.` / `Relatório excluído.` (singular past participle, matching `Receita excluída.` / `Atestado excluído.`).

## Surface-specific interaction contracts

These are the only places the new docs differ from a straight prescription clone. Each still lives inside the shipped wizard/list shell.

**DOC-01 Encaminhamento** — wizard identical to prescription-wizard (patient/manual/template steps). Fields: specialty/service = `Combobox` (`components/ui/combobox.tsx`) seeded with common pediatric destinations (ORL, oftalmo, neuroped, fono, fisio, nutri…) + free-text "outro"; motivo = `Textarea`; resumo clínico/hipótese = `Textarea`; urgência = `Select` with three items `Rotina` / `Prioritário` / `Urgente` (default `Rotina`). Urgency renders as the semantic `Badge` on the list/card (see Color).

**DOC-02 Pedido de exames** — searchable catalog: reuse the normalize/filter pattern from `report-template-search-input.tsx` (simple filtered list is the lower-risk default; `cmdk`/`command.tsx` optional). Selecting an exam appends an **editable line item**; a free-text field adds off-catalog items; applying a panel expands into the same editable item list (D-03) — items are removable via `Trash2` ghost icon-button with `aria-label="Remover exame"`, mirroring the medication-row remove control in prescription-wizard. Plus hipótese + observações `Textarea`s. Empty item list uses the dashed inline hint `Nenhum exame adicionado. Busque acima ou digite um exame.` (matches `Nenhum medicamento adicionado…`).

**DOC-03 Relatório médico** — título/finalidade `Input` + single `RichTextEditor` free body (NOT sections). Toolbar/placeholder come from the shipped `rich-text-editor.tsx`; placeholder e.g. `Escreva o relatório…`. No multi-section UI, no import from `report-templates`.

**DOC-05 Receituário em branco** — the existing prescription surface in a "blank/vazio" mode (`?mode=blank` on the existing new-prescription route). Same layout, empty medication list/body, same CTA styling but label `Gerar receituário`. Zero new visual surface — visual parity is guaranteed by reusing the component.

**DOC-06 Orientações** — own route: pick a puericultura milestone (1ª consulta, 1m, 2m, 4m, 6m, 9m, 12m, 18m, 24m, anual, + doctor-added), the milestone's editable guidance text loads into a preview, patient header auto-fills, then `Gerar orientação` → PDF. Milestone selection via `Select` or a compact list; content is doctor-editable seed (human-verify checkpoint — content, not UI).

**Auto-filled patient header (all docs):** nome / DOB / idade, where idade uses `computePediatricAge` (Phase-1 engine) rendered in the PT-BR band string. Entry from patient profile pre-selects (`?patientId=`/`?caseId=`); standalone uses `MedicalCertificatePatientPickerSheet` (D-13). Header block styling matches the shipped wizard patient block (`Field`/`FieldLabel` + muted secondary lines).
